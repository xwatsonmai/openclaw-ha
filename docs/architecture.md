# 架构规划

## 目标

让“智能管家”在回答高频状态问题时，优先读取本地同步快照，而不是每次临时访问 Home Assistant。

## 为什么不把实时状态直接写入提示词

不建议把动态状态直接注入默认提示词，原因：

1. 状态变化快，容易过期
2. 提示词体积会不断膨胀
3. 每次改提示词的成本高，不适合高频更新
4. 数据与规则混在一起，不利于维护

更好的模式是：
- 提示词：只描述读取规则与决策规则
- 本地快照：保存动态状态
- 回答逻辑：先读快照，必要时再实时查询

## 推荐架构

### 1. 数据采集层
两种来源：

- 定时轮询（第一版先做）
- 事件驱动（后续增强）
  - HA WebSocket 订阅 `state_changed`
  - 或 HA 自动化调用 webhook

### 2. 状态存储层
保存两类文件：

#### 原始状态
- 文件：`data/raw_states.json`
- 用途：保留 HA 返回的主要实体状态，便于排查与后处理

#### 摘要状态
- 文件：`data/summary.json`
- 用途：直接服务回答
- 示例字段：
  - 生成时间
  - 数据新鲜度
  - 灯光概览
  - 空调概览
  - 环境信息
  - 离线设备
  - 最近异常

### 3. 回答决策层
默认回答逻辑：

1. 先读 `summary.json`
2. 若数据足够新，则直接回答
3. 若缓存过期、问题高风险、或用户明确要求实时，则临时查询 HA
4. 控制设备后，立即刷新相关实体或触发一次同步

## 哪些问题适合直接读缓存

适合：
- 家里现在什么情况
- 哪些灯开着
- 空调现在开没开
- 室内温度多少
- 最近有没有离线设备

不应只靠缓存：
- 立刻刷新一下现在状态
- 刚刚谁触发了门磁
- 门锁现在是不是刚打开
- 安防/供电/门锁等高风险状态确认

## 第一版最小实现

第一版先做：

1. 复用现有 HA 配置
2. 拉取全部实体状态
3. 生成本地快照
4. 输出一个简化摘要

先不做：
- 守护进程
- webhook 服务
- WebSocket 常驻订阅
- 自动写回智能管家的规则文件

## 第二版建议

1. 关键实体白名单
2. 房间维度摘要
3. 离线设备告警
4. 事件日志
5. WebSocket 实时订阅
6. 同步失败重试与日志归档

## 推荐最终方案

采用：

**事件驱动为主 + 定时全量同步兜底 + 本地摘要缓存回答**

这样兼顾：
- 实时性
- 稳定性
- 简洁性
- 回答速度
